Issue #03 · API: Auth
Sessions Endpoint Returns 401 — Bearer Token Rejected
GET /api/auth/sessions
High

🔴 What's Wrong

The /api/auth/sessions endpoint only accepts cookie-based authentication (httpOnly session cookies). However, the admin panel at /admin/sessions authenticates using JWT Bearer tokens stored in localStorage. This mismatch causes the page to always receive a 401, making session management completely inaccessible from the admin interface.

✅ How to Fix It

  • Update the /api/auth/sessions route to accept both Bearer tokens (from Authorization header) and httpOnly cookies
  • The requireRole() auth middleware should check header first, then fall back to cookie
  • Alternatively, move session data into a separate admin-specific endpoint like /api/admin/sessions that already uses Bearer auth
// lib/auth.ts — accept both auth methods export function requireRole(req: NextRequest, roles: string[]) { const bearer = req.headers.get('authorization')?.replace('Bearer ', ''); const cookie = req.cookies.get('session_token')?.value; const token = bearer || cookie; if (!token) return unauthorized(); return verifyToken(token, roles); }

💡 What It Does When Fixed

Admins can view all active user sessions across the platform — who is logged in right now, from which IP, on which device, and for how long. They can forcibly terminate any session (e.g., to lock out a compromised account). This is a critical security feature for platform governance.

Issue #14 · Role: Admin
Accepted Shops Page — JS Crash: undefined.replace() TypeError
/admin/accepted-shops
Critical

🔴 What's Wrong

The page crashes with TypeError: Cannot read properties of undefined (reading 'replace') inside a sort function. The shop list API returns shops where at least one shop has a null or undefined name field. The sort comparator calls name.replace() without a null check, which throws and crashes the entire React component tree, showing an error boundary instead of the page.

// Broken sort — crashes on null name shops.sort((a, b) => a.name.replace(...) > b.name.replace(...)) // ↑ throws if a.name or b.name is undefined

✅ How to Fix It

  • Add a null-safe sort comparator using optional chaining and fallback strings
  • Also add a null guard in the API response to never return undefined name fields
// Fixed sort — null-safe shops.sort((a, b) => { const nameA = (a.name || '').toLowerCase(); const nameB = (b.name || '').toLowerCase(); return nameA.localeCompare(nameB); });

💡 What It Does When Fixed

The /admin/accepted-shops page renders correctly for all admins. They can see a full list of every approved shop on the platform — sortable by name, creation date, or revenue — with links to each shop's detail page. Admins can then take actions like suspending a shop, viewing metrics, or checking subscription status. This is one of the most frequently used admin pages.
