A and B: Customer-Created Flows

  1. Customer opens appointments flow at /customer/appointments/new.
  2. NewAppointmentClient selects shop, visit type (in-shop or road-call), service, date/details, optional media.
  3. Submit calls POST /api/appointments.
  4. API creates appointment plus linked workOrder with status pending and serviceLocation from appointmentType.
  5. Customer can monitor through detail route /customer/workorders/[id] and payment list /customer/payments.

C and D: Shop-Employee-Created Flows

  1. In-shop: /workorders/inshop routes to /shop/new-inshop-job, then submit calls POST /api/workorders.
  2. Road-call: /workorders/roadside routes to /shop/new-roadside-job.
  3. Current road-call submit does not call a create API and only navigates back home.
  4. Result: in-shop creates real work orders; roadside UI currently acts as a draft-only path.

Completion and Payment Steps

  1. Shop updates work order state through PUT /api/workorders/[id] across pending, assigned, in-progress, waiting-estimate, waiting-for-payment, closed.
  2. When set to waiting-for-payment, customer receives completion and payment-ready notifications.
  3. Customer pays from /customer/workorders/[id] or /customer/payments via POST /api/payment/checkout.
  4. Stripe returns to /payment/success or /payment/cancel.
  5. Stripe webhook marks work order paid and closed.

What Work Order Does After Payment

  1. Status and payment fields update to closed and paid.
  2. Customer gets receipt email and push confirmation.
  3. Customer sees closed confirmation on work order detail and payment pages.
  4. Invoice generation and email endpoints are available for retrieval and sending.
  5. If closure happens through PUT /api/workorders/[id], extra side effects run: loyalty points, review request email/SMS, and inventory deduction.
